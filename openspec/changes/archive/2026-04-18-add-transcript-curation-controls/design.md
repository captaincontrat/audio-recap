## Context

The earlier transcript changes now establish three prerequisites:

- `add-web-meeting-processing` defines the durable transcript record, canonical `transcriptMarkdown` and `recapMarkdown`, and the processing-owned transcript `title`.
- The reduced `add-transcript-management` change defines the transcript library and detail read surfaces, including the stable `displayTitle` read contract, baseline search and sorting, cursor pagination, and explicit loading and error states.
- `add-transcript-edit-sessions` now owns markdown edit sessions, locking, autosave, expiration, and conflict handling for transcript content editing.

This change layers workspace-scoped metadata curation onto those transcript surfaces. It does not redefine read-only transcript browsing, markdown edit sessions, public sharing, or export. Instead, it defines how workspace members and admins can rename, organize, and delete transcript records that already belong to the current workspace.

## Goals / Non-Goals

**Goals:**

- Define workspace-scoped transcript rename, tag management, important toggle, and delete confirmation behavior.
- Preserve the processing-owned `title` while defining the curation-owned effective-title rule `customTitle ?? title` that feeds the stable `displayTitle` read contract owned by `add-transcript-management`.
- Define the role-based authorization and validation rules for transcript curation writes.
- Extend the workspace transcript library with important-state and tag-aware organization controls.

**Non-Goals:**

- Markdown editing, autosave, lock acquisition or expiry, or conflict handling; those belong to `add-transcript-edit-sessions`.
- Public sharing URLs and share management.
- Export formats or frontend conversion from markdown.
- Changes to transcript generation, retention, or background processing behavior beyond what curation needs to respect.
- The stable transcript library/detail `displayTitle` read contract; that remains with `add-transcript-management`.
- Workspace archival policy beyond honoring the active-workspace gate from `add-workspace-archival-lifecycle`.
- Collaborative comments, version history, or audit timelines.
- Rich text or WYSIWYG storage formats.

## Decisions

### Decision: Preserve the processing-owned title and add a nullable `customTitle` override

`add-web-meeting-processing` already defines a durable transcript `title` generated by processing. `add-transcript-management` owns the stable read-side `displayTitle` contract consumed by the library and detail surfaces. This change keeps the processing-owned field as the system-owned baseline and adds:

- `customTitle`: nullable user override written by curation surfaces

This change owns the effective-title calculation rule `customTitle ?? title`, which feeds the stable `displayTitle` read contract without moving read-surface ownership out of `add-transcript-management`.

This lets workspace members and admins rename any transcript record in the current workspace without losing the generated baseline and without forcing the worker to coordinate with user-driven mutations.

**Why this over alternatives**

- Over mutating the processing-owned `title` in place: that loses the system-generated baseline and creates worker or replay overwrite races.
- Over introducing a second generated-title field after processing already chose `title`: it would add avoidable schema churn for the same product outcome.
- Over letting this change own the durable read-side title contract: the library and detail surfaces should keep their stable read contract in `add-transcript-management`.

### Decision: Use one transcript patch surface for non-destructive metadata updates

The curation API will use:

- the existing read surfaces from reduced transcript management
- one patch surface for rename, tags, and important toggle
- one delete surface for destructive removal

The patch surface accepts only workspace-authorized metadata fields:

- `customTitle`
- `tags`
- `isImportant`

Validation rules:

- `customTitle`, `tags`, and `isImportant` are editable for any transcript in the current workspace by a user with workspace role `member` or `admin`
- tags are normalized, deduplicated, and limited in count and length
- `transcriptMarkdown` and `recapMarkdown` are intentionally excluded here and handled by `add-transcript-edit-sessions`

**Why this over alternatives**

- Over separate endpoints for every small toggle: one patch surface keeps the API simpler without weakening role-based controls.
- Over keeping markdown edits in the same patch contract: the edit-session lock and autosave rules make markdown editing a materially different concern.

### Decision: Tags are a normalized per-transcript string list

Tags are a lightweight organizational feature, not a first-class workspace taxonomy. The durable transcript record will store tags as a normalized set of strings:

- case-insensitive uniqueness
- trimmed values
- normalized casing for storage and filtering
- bounded maximum count and per-tag length

The backend may also derive a deterministic normalized tag sort key from the sorted tag list to support stable tag-aware organization in the library.

**Why this over alternatives**

- Over a separate global tag table: that is heavier than necessary for the first workspace transcript curation pass.
- Over unbounded free-form tags: normalization and limits prevent noisy duplicates and poor filter UX.

### Decision: Important and tag-aware organization extend the workspace transcript library

The reduced transcript library already owns baseline search, status filtering, and `displayTitle` or time sorting. This curation change adds metadata-rich organization controls on top of that base:

- important-first and important-last sorts
- tag-list A-Z and tag-list Z-A sorts
- important-state filtering
- selected-tag filtering

For tag-based sorting, each transcript record uses a deterministic normalized tag sort key derived from its sorted normalized tag list. Records without tags sort after tagged records for ascending tag-list order and before tagged records for descending tag-list order.

**Why this over alternatives**

- Over pushing all organization behavior into the first library change: it makes the initial transcript foundation too large.
- Over keeping tags and important markers visible only inside per-record controls: users need those signals to organize the workspace transcript library.

### Decision: Delete is permanent and uses workspace roles plus creator attribution

A workspace `member` may permanently delete only transcripts they created while the transcript still retains creator attribution that resolves to that member's current account. A workspace `admin` may permanently delete any transcript in the current workspace, including retained records whose creator account was later permanently deleted. The frontend must require explicit confirmation before the delete request is sent.

After successful deletion:

- the record no longer appears in the workspace transcript library
- direct detail reads resolve to the same not-found behavior used for missing or out-of-workspace records

This change does not define undo or trash behavior.

**Why this over alternatives**

- Over soft delete in the first iteration: it adds lifecycle complexity that the current roadmap does not require.
- Over creator-only deletion for all roles: the workspace collaboration model explicitly gives admins broader cleanup authority.
- Over silent destructive controls: transcript deletion is high-risk enough to require confirmation.
- Over letting any workspace `member` delete a retained record once creator attribution has been cleared: destructive authority should fall back to `admin` once there is no surviving creator account to match against.

### Decision: Enforce workspace scoping for curation writes and hide out-of-workspace records

Every transcript-curation operation will be scoped by the current workspace and its membership rules. For workspace-scoped private curation routes, the explicit workspace route context from `add-workspace-foundation` is authoritative; session or remembered state may help choose a default entry route when no workspace is explicit, but must not override the route-selected workspace. `add-workspace-archival-lifecycle` owns the rule that archived workspaces are inactive for collaboration, so these mutation surfaces depend on that active-workspace gate rather than redefining archive policy. Direct patch or delete access to a transcript identifier outside the current workspace returns the same not-found behavior as a missing transcript so the system does not reveal record existence across workspaces.

Inside the current workspace, mutation permissions follow role rules:

- `member` and `admin` can patch `customTitle`, `tags`, and `isImportant`
- `member` can delete only transcripts whose creator attribution still resolves to that member
- `admin` can delete any transcript in the workspace

This workspace model applies only to authenticated transcript records. Future public share routes remain out of scope here.

**Why this over alternatives**

- Over returning explicit forbidden errors for out-of-workspace records: hiding existence is a safer default across workspace boundaries.
- Over keeping curation owner-only inside a workspace: that would fight the shared-workspace collaboration model.
- Over allowing curation writes to continue while a workspace is archived: archived workspaces are inactive for collaboration under `add-workspace-archival-lifecycle`.

### Decision: Curation controls extend the existing transcript surfaces instead of creating a separate management area

Rename, tags, important, and delete actions belong on top of the transcript library and detail surfaces rather than in a separate transcript-admin area.

This means:

- the library can expose lightweight organization signals and quick actions where appropriate
- the detail page can host the full rename, metadata, and delete flows
- markdown editing can later layer onto the same detail surface through `add-transcript-edit-sessions`
- later sharing and export controls can continue to layer onto the same transcript surfaces

**Why this over alternatives**

- Over creating a separate management area: it would fragment one workspace transcript workflow across too many product surfaces.

## Risks / Trade-offs

- [User metadata changes can race with processing-owned title generation] -> Keep the generated baseline in `title`, store user overrides separately in `customTitle`, and compute the display title from both.
- [Important and tag-aware organization increase query complexity] -> Derive stable sort helpers and keep the read/write boundary explicit between the reduced library base and this curation extension.
- [Delete is irreversible and permissions depend on both role and creator-attribution state] -> Require explicit confirmation and enforce creator or admin rules on the server.
- [Splitting metadata curation from markdown edit sessions can feel fragmented] -> Keep both changes layered onto the same transcript library and detail surfaces instead of separate product areas.

## Migration Plan

1. Extend transcript persistence to support `customTitle`, tags, important state, and any supporting derived sort or search fields needed for curation queries.
2. Add workspace-scoped transcript patch and delete surfaces with validation for rename, tags, important toggles, creator or admin delete rules including deleted-creator fallback behavior, workspace not-found behavior, and the active-workspace gate from `add-workspace-archival-lifecycle`.
3. Extend the workspace transcript library with important and tag-aware organization controls.
4. Add curation UI on top of the transcript library and detail surfaces for rename, tags, important toggle, and delete confirmation, while keeping markdown edit entry and save flows in `add-transcript-edit-sessions`.
5. Add regression coverage for active-workspace lockout plus workspace-scoped curation mutations, normalization, destructive actions, and important or tag-aware organization behavior.

Rollback strategy:

- keep transcript creation and read surfaces intact even if curation controls are disabled
- disable new mutation controls and endpoints if needed without affecting processing or transcript browsing
- preserve the new metadata fields in Postgres even if the curation UI is temporarily withdrawn

## Open Questions

None are blocking for this change. Markdown editing, autosave, lock expiry, and related conflict behavior remain separate in `add-transcript-edit-sessions`, while sharing and export remain separate follow-up changes.
