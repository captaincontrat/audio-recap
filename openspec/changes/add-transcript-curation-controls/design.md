## Context

The earlier transcript changes now establish two prerequisites:

- `add-web-meeting-processing` defines the durable transcript record, canonical `transcriptMarkdown` and `recapMarkdown`, and the processing-owned transcript `title`.
- The reduced `add-transcript-management` change defines the private transcript library and detail read surfaces, including owner-scoped list and detail access, baseline search and sorting, cursor pagination, and explicit loading and error states.

This change layers owner-driven curation onto those private surfaces. It does not redefine read-only transcript browsing, public sharing, or export. Instead, it defines how owners can rename, correct, organize, and delete transcript records they already own.

## Goals / Non-Goals

**Goals:**

- Define owner-driven transcript rename, markdown editing, tag management, important toggle, and delete confirmation behavior.
- Keep the backend-to-frontend contract markdown-first for transcript and recap content.
- Define the write-side ownership and validation rules for transcript curation.
- Extend the private transcript library with important-state and tag-aware organization controls.

**Non-Goals:**

- Public sharing URLs and share management.
- Export formats or frontend conversion from markdown.
- Changes to transcript generation, retention, or background processing behavior beyond what curation needs to respect.
- Collaborative editing, comments, version history, or audit timelines.
- Rich text or WYSIWYG storage formats.

## Decisions

### Decision: Preserve the processing-owned title and add a nullable `customTitle` override

`add-web-meeting-processing` already defines a durable transcript `title` generated by processing. This change keeps that field as the system-owned baseline and adds:

- `customTitle`: nullable user override written by curation surfaces

The user-facing display title is computed as `customTitle ?? title`.

This lets users rename any transcript record without losing the generated baseline and without forcing the worker to coordinate with user-driven mutations.

**Why this over alternatives**

- Over mutating the processing-owned `title` in place: that loses the system-generated baseline and creates worker or replay overwrite races.
- Over introducing a second generated-title field after processing already chose `title`: it would add avoidable schema churn for the same product outcome.

### Decision: Use one transcript patch surface for non-destructive updates

The curation API will use:

- the existing read surfaces from reduced transcript management
- one patch surface for rename, markdown edits, tags, and important toggle
- one delete surface for destructive removal

The patch surface accepts only owner-authorized fields:

- `customTitle`
- `transcriptMarkdown`
- `recapMarkdown`
- `tags`
- `isImportant`

Validation rules:

- `transcriptMarkdown` and `recapMarkdown` are editable only when the transcript is in `completed` status
- `customTitle`, `tags`, and `isImportant` are editable for any owned record
- tags are normalized, deduplicated, and limited in count and length

**Why this over alternatives**

- Over separate endpoints for every small toggle: one patch surface keeps the API simpler without weakening ownership controls.
- Over allowing markdown edits on non-completed records: those records may not yet have stable canonical content.

### Decision: Tags are a normalized per-transcript string list

Tags are a lightweight organizational feature, not a first-class collaborative taxonomy. The durable transcript record will store tags as a normalized set of strings:

- case-insensitive uniqueness
- trimmed values
- normalized casing for storage and filtering
- bounded maximum count and per-tag length

The backend may also derive a deterministic normalized tag sort key from the sorted tag list to support stable tag-aware organization in the library.

**Why this over alternatives**

- Over a separate global tag table: that is heavier than necessary for private single-owner transcript curation.
- Over unbounded free-form tags: normalization and limits prevent noisy duplicates and poor filter UX.

### Decision: Important and tag-aware organization extend the private library

The reduced private library already owns baseline search, status filtering, and title/time sorting. This curation change adds metadata-rich organization controls on top of that base:

- important-first and important-last sorts
- tag-list A-Z and tag-list Z-A sorts
- important-state filtering
- selected-tag filtering

For tag-based sorting, each transcript record uses a deterministic normalized tag sort key derived from its sorted normalized tag list. Records without tags sort after tagged records for ascending tag-list order and before tagged records for descending tag-list order.

**Why this over alternatives**

- Over pushing all organization behavior into the first library change: it makes the initial transcript foundation too large.
- Over keeping tags and important markers visible only inside per-record controls: users need those signals to organize their private library.

### Decision: Delete is permanent and requires explicit confirmation

The owner of a transcript record may permanently delete it. The frontend must require explicit confirmation before the delete request is sent.

After successful deletion:

- the record no longer appears in the private library
- direct detail reads resolve to the same not-found behavior used for missing records

This change does not define undo or trash behavior.

**Why this over alternatives**

- Over soft delete in the first iteration: it adds lifecycle complexity that the current roadmap does not require.
- Over silent destructive controls: transcript deletion is high-risk enough to require confirmation.

### Decision: Enforce ownership for all curation writes and hide non-owned records

Every transcript-curation operation will be scoped by `ownerUserId`. A user can only patch or delete transcript records they own. Direct patch or delete access to another user's transcript identifier returns the same not-found behavior as a missing transcript so the system does not reveal record existence across accounts.

This ownership model applies only to authenticated private transcript records. Future public share routes remain out of scope here.

**Why this over alternatives**

- Over returning explicit forbidden errors for known foreign records: hiding existence is a safer default for private resources.

### Decision: Curation controls extend the existing private surfaces instead of creating a separate management area

Rename, tag, important, markdown editing, and delete actions belong on top of the private transcript library and detail surfaces rather than in a separate transcript-admin area.

This means:

- the library can expose lightweight organization signals and quick actions where appropriate
- the detail page can host the full markdown-first editing and delete flows
- later sharing and export controls can continue to layer onto the same private transcript surfaces

**Why this over alternatives**

- Over creating a separate management area: it would fragment one owner workflow across too many product surfaces.

## Risks / Trade-offs

- [User edits can race with processing-owned title generation] -> Keep the generated baseline in `title`, store user overrides separately in `customTitle`, and compute the display title from both.
- [Markdown editing can feel technical for some users] -> Keep the persistence contract markdown-first now and allow richer editing affordances later without changing storage.
- [Important and tag-aware organization increase query complexity] -> Derive stable sort helpers and keep the read/write boundary explicit between the reduced library base and this curation extension.
- [Delete is irreversible] -> Require explicit confirmation and keep owner-scoped not-found behavior consistent after deletion.

## Migration Plan

1. Extend transcript persistence to support `customTitle`, tags, important state, and any supporting derived sort or search fields needed for curation queries.
2. Add owner-scoped transcript patch and delete surfaces with validation for rename, markdown edits, tags, important toggles, and delete confirmation flows.
3. Extend the private transcript library with important and tag-aware organization controls.
4. Add curation UI on top of the private library and detail surfaces for rename, tags, important toggle, markdown editing, and delete confirmation.
5. Add regression coverage for owner-scoped curation mutations, normalization, destructive actions, and important or tag-aware organization behavior.

Rollback strategy:

- keep transcript creation and private read surfaces intact even if curation controls are disabled
- disable new mutation controls and endpoints if needed without affecting processing or private transcript browsing
- preserve the new metadata fields in Postgres even if the curation UI is temporarily withdrawn

## Open Questions

None are blocking for this change. Public sharing and export remain separate follow-up changes that can add more owner actions to the same private transcript surfaces.
